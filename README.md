## 💡 2. CPU 스케줄링 알고리즘 한계 및 CFS 필요성 분석

### 2.1 기존 스케줄링 방식의 한계 (복합문제 분석)
* SJF / SRTF: 대기시간 최소화에는 좋으나 프로세스 남은 시간을 정확히 예측할 수 없어 비현실적임.
* FCFS / Priority: 긴 작업 뒤에 짧은 작업이 밀리는 호위 효과(Convoy Effect)가 발생하거나, 낮은 우선순위 작업이 영원히 실행되지 못하는 기아(Starvation) 현상이 발생함.
* Round-Robin (RR): 공평성을 위해 시간을 쪼개지만, 주기가 짧을수록 컨텍스트 스위칭 오버헤드가 급증함.

### 2.2 리눅스 CFS의 도입 필요성
기존 알고리즘의 딜레마(공평성 보장 vs 오버헤드 증가)를 극복하기 위해 CFS가 도입됨.
1. vruntime(가상 실행 시간) 기반의 공평성: 무조건 CPU를 가장 적게 쓴 프로세스에게 CPU를 넘겨주어 기아 현상을 완벽히 차단함.
2. 동적 실행 시간 할당: 프로세스 개수와 가중치에 맞춰 시간을 동적으로 배분해 무의미한 문맥 교환(Context Switching) 오버헤드를 막아줌.
3. Red-Black Tree 도입: 실행 시급성이 높은 프로세스를 O(log N)만에 찾아내어 스케줄러 자체 성능을 극대화함.
